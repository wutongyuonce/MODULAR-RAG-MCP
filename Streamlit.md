Streamlit 是一个能将 Python 脚本快速转化为交互式 Web 应用的开源框架。它的核心机制简洁而强大，让数据科学家和开发者无需前端知识也能轻松构建和分享数据应用。

下面是一份从入门到基本了解的完整笔记，覆盖了其核心机制、用法与最佳实践。

### 🌐 Streamlit 的核心机制：Python 脚本即交互式 Web 应用

Streamlit 的精髓在于其“脚本即应用”的执行模型。它的核心工作流程和设计理念主要围绕以下几点展开：

#### 🔄 执行模型

- **全量重新执行**：这是 Streamlit 最核心的运行逻辑。当用户与应用交互（如滑动滑块、点击按钮）时，整个 Python 脚本会从头到尾**完整地重新运行**。这种看似“笨拙”的设计，实则极大地简化了开发者的心智负担，因为你无需构建复杂的前端事件监听和回调函数。
- **数据流与状态管理**：
  - **无状态设计 (Stateless)**：每次脚本重新执行时，普通变量都会被重置。默认行为下，只有 UI 组件 (`st.slider`, `st.text_input` 等) 的状态会被 Streamlit 自动保留，并在下次执行时注入最新的值。
  - **`st.session_state`**: 为了应对需要跨运行保存数据（如用户登录状态、购物车信息）的场景，Streamlit 提供了`st.session_state`。它是一个字典，可以在不同的脚本执行间持久化存储任何你想要的数据。

#### ⚙️ 缓存机制 (`st.cache_data`)

全量重新执行可能会导致重复进行耗时的计算（如加载大型数据集、训练模型）。为此，Streamlit 提供了缓存装饰器来优化性能。

- **`@st.cache_data`**: 首选推荐。它会根据函数的输入参数对结果进行缓存。当函数被再次调用且参数未变时，直接返回缓存结果，跳过重计算。这对于处理DataFrame、NumPy数组等可序列化数据的函数非常有效。
- **`@st.cache_resource`**: 专用于缓存全局资源，例如已加载的机器学习模型或数据库连接。这些对象通常体积大且不可序列化，使用该装饰器可以避免在每次重新运行时重复加载，极大地提升效率。

#### 🏗️ 应用架构

Streamlit 应用采用标准的**客户端-服务器**架构。

- **服务器端**：运行你的 Python 脚本，负责所有数据处理和计算的核心逻辑。
- **客户端**：用户的浏览器。它接收用户操作，并通过一个由 Tornado 框架支持的**WebSocket**与服务器端建立**持久、双向的通信**。每当用户交互时，信息会通过 WebSocket 发送回服务器，触发脚本重新执行，服务器再将更新后的 UI 渲染指令推送给前端。

#### 🎨 前端集成与扩展

- **内置组件**：Streamlit 提供了丰富的内置 UI 组件，如`st.write`、`st.button`、`st.dataframe`等，这些组件被 Streamlit 自动渲染为现代、美观的网页元素。
- **自定义组件**：当内置组件无法满足需求时，可以创建**自定义组件**。它本质上是一个独立的前端模块（通常使用 React 构建），能与 Python 后端进行双向通信，从而极大地扩展了 Streamlit 的能力。
- **主题与样式**：Streamlit 支持通过配置文件 `.streamlit/config.toml` 进行主题定制，包括主色、背景色、字体等，也可以使用自定义 CSS 来微调页面样式。
